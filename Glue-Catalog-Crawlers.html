¿Dónde encaja Glue?

  • S3 guarda los archivos. RDS guarda las transacciones.
  • Glue es el «traductor» que entiende qué hay adentro y lo deja listo para analizar.
  • Hoy: Data Catalog + Crawlers + ETL sobre S3.
teraclud

Almacenar vs catalogar

S3 (almacenar)

  • Archivos CSV, JSON, Parquet, etc.
  • Durabilidad y bajo costo por GB.
  • Sin esquema obligatorio al escribir.
  • Como el depósito del data lake.

Glue Catalog (catalogar)

  • Metadatos: tablas, columnas, tipos.
  • Ubicación S3 de cada tabla.
  • Particiones detectadas o definidas.
  • Sin catálogo, Athena no sabe qué leer.
teraclud

AWS Glue en 30 segundos

  • Servicio serverless de integración de datos en AWS.
  • Glue Data Catalog: inventario central de metadatos (tablas, columnas, particiones).
  • Crawlers: escanean S3 y alimentan el catálogo automáticamente.
  • ETL Jobs: transforman datos (típicamente S3 → S3). El job estándar usa Spark (PySpark); también existe Python Shell para scripts livianos en Python puro.
  • Se integra con Athena, Lake Formation, Redshift Spectrum y más.
teraclud

El catálogo: tu mapa del tesoro

  • Sin catálogo, S3 es una carpeta con 10.000 archivos y cero contexto.
  • Tablas = punteros a datos en S3 + esquema + particiones.
  • Una base Glue = un namespace (ej.: workshop_glue).
  • Athena, Redshift Spectrum y otros servicios consultan el mismo catálogo — un solo inventario de metadatos.
  • Podés registrar tablas a mano en el catálogo; el crawler solo automatiza el descubrimiento inicial.
teraclud

Crawlers: el detective que a veces se equivoca

  • Recorren un path S3, infieren esquema y crean/actualizan tablas.
  • Config clave: data source, target database, frecuencia manual.
  • Una carpeta por entidad → una tabla por dataset (bookings, customers, etc.).
  • Ojo: pueden inferir STRING donde hay fechas, o ignorar particiones mal nombradas.
  • Es como el autocorrect del teléfono: te ahorra tiempo, pero conviene releer antes de publicar.
teraclud

Athena: consultar sin mover el dato

  • Athena lee directamente desde S3 usando el catálogo — no importa los datos a otro lado.
  • Eso es schema-on-read: el esquema se aplica al consultar.
  • Configurá siempre athena-results/ para los outputs de las queries.
  • Pagás por datos escaneados — una query sin filtros se nota en el costo.
  • Sirve para explorar rápido y validar que el catálogo refleja lo que hay en S3.
teraclud

Schema-on-read vs schema-on-write

Schema-on-read (Data Lake)

  • Los datos llegan como vienen (CSV, JSON, logs).
  • El esquema se aplica al leer (Athena, Glue).
  • Flexible, rápido para ingerir.
  • Riesgo: inconsistencias si no gobernáis.

Schema-on-write (Warehouse)

  • Definís tablas antes de cargar.
  • Datos rechazados si no cumplen.
  • Más riguroso para BI clásico.
  • Glue curated suele ser el punto medio.
teraclud

Particiones: el truco para no escanear medio planeta

  • Estructura: s3://bucket/raw/ventas/year=2025/month=06/day=30/...
  • Athena y Glue pueden «podar» particiones: solo leen lo que pedís.
  • Query sin filtro de partición → escanea todo → más costo.
  • Las carpetas en S3 no existen «de verdad» — son prefijos. Pero para el catálogo, sí importan.

¿Glue, Lambda o EMR?

Usá Glue cuando…

  • Transformaciones batch sobre el Data Lake.
  • Querés catálogo + ETL integrado.
  • Trabajás con Parquet y volúmenes medianos/grandes.
  • No querés administrar un cluster.

Otro tool cuando…

  • Lambda: archivos chicos, evento por archivo, lógica liviana.
  • EMR: Spark a escala custom, notebooks, jobs muy pesados.
  • DMS / App Flow: replicar bases, no transformar el lake.
  • No hay bala de plata — hay herramienta menos incómoda.

Anatomía de un job ETL (sin entrar al sótano)

  • Input/Source: de dónde lee (hoy: S3 en raw/).
  • Transform: filtrar, limpiar, castear, deduplicar, agregar columnas.
  • Output/Sink: dónde escribe (hoy: S3 en curated/ en Parquet).
  • El job ETL estándar corre Spark por detrás — vos no administrás el cluster.
  • Alternativa: Python Shell para transformaciones en Python puro (pandas, boto3) sin Spark.
  • Otras opciones en el ecosistema: Lambda (archivos chicos) o EMR (Spark a escala custom).
  • Existen otras configuraciones específicas como: conexiones JDBC, job bookmarks y triggers por eventos de S3.
teraclud

Transformaciones típicas (curated layer)

  • Renombrar columnas a un estándar (snake_case, sin espacios raros).
  • Castear tipos: string → date, string → int donde corresponda.
  • Filtrar filas basura (nulls críticos, fechas imposibles, duplicados).
  • Enmascarar o excluir columnas PII (datos personales sensibles) según política.
  • Escribir en el formato que necesites: .parquet, particionado, compresión, etc.
teraclud

Calidad de datos: el plot twist del ETL

  • Transformar no es solo «cambiar de formato» — es confiar en lo que sale.
  • Checks simples: conteos antes/después, % nulls, rangos de fechas.
  • Si el curated tiene más filas que el raw, algo raro pasó (spoiler: duplicados).
  • Cuidado con JOINs que duplican métricas al unir tablas con relaciones 1 a N.
  • Generar documentación — tu yo del futuro te lo agradecerá.

Checklist de buenas prácticas

  • Una carpeta por entidad en raw/ — no mezclar CSV de distintas tablas.
  • raw = como llegó; curated = limpio y tipado; analytics = listo para consumo; no mezcles capas.
  • Crawler para descubrir; job ETL para corregir de verdad.
  • Para revisar ejecuciones de queries. Configurar athena-results/.
  • Documentá en inglés.
  • Borrá recursos al terminar.
